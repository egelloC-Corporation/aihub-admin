I need to deploy my app on Incubator at incubator.egelloc.com. The platform handles SSO, port assignment, and deployment automatically. Walk me through the setup. ## Platform info - Apps are served at incubator.egelloc.com/APP_SLUG/ behind a reverse proxy (nginx). The proxy strips the /APP_SLUG prefix before forwarding to the container, so the app sees root-relative paths (/, /api/..., etc.). - Auth is handled by the platform's Google OAuth SSO. Apps verify sessions by calling AIHUB_AUTH_URL (set at runtime, reads the platform session cookie). - Ports are auto-assigned — I do NOT need to pick one. The deploy system assigns a free port and sets the PORT env var. - After approval, every push to the GitHub main branch auto-deploys. ## What I need you to do 1. Add the auth SDK from https://github.com/egelloc-corporation/aihub-admin/tree/main/auth-sdk — use aihub-auth.js (Node) or aihub_auth.py (Python, needs `requests`). Wire it into my routes: - loginRequired for general auth - requirePermission("APP_SLUG") for app-specific access - The SDK reads AIHUB_AUTH_URL from the environment at runtime. Do NOT bake AIHUB_AUTH_URL into build-time config (e.g. next.config.ts env{}) — it will be empty during docker build and override the real runtime value. 2. Add a /health endpoint that returns { "status": "ok" } 3. If my app serves HTML, add this before </body> on every page: <script src="/hub-navbar.js" defer></script> This script does three things: loads the Incubator app-switcher drawer, injects the platform favicon, and renders the platform header banner (egg + "Incubator" wordmark + divider + app name) at the top of the page. Clicking the wordmark goes to /launcher; clicking the app name goes to the app's own root. PLATFORM CHROME CONTRACT — what the platform owns vs. what your app owns: - Do NOT render your own "Incubator" brand or your own app name in your header — the banner already shows them. - Do NOT add <link rel="icon"> tags — the script manages favicons centrally. - DO put app-specific controls (selectors, filters, tabs, search) in your own sub-header below the banner, or integrated into your main view. - CSS: the banner is sticky at top with height 48px (44px on mobile). If you use `position: sticky; top: 0` inside your app, change it to `top: 48px` so it stacks below the banner. - Full spec: docs/platform-banner.md 4. CRITICAL — handle the path prefix. The app is served at /APP_SLUG/ but receives requests at / (nginx strips the prefix). The fix depends on the stack: Flask / Jinja / vanilla HTML: - Use RELATIVE paths in templates: href="static/css/style.css" NOT href="/static/css/style.css" - Use RELATIVE fetch paths in JS: fetch("api/stats") NOT fetch("/api/stats") - Any URL starting with "/" bypasses the app and hits the platform admin panel Next.js: - Set basePath in next.config.ts reading from APP_SLUG env var: basePath: process.env.APP_SLUG ? `/${process.env.APP_SLUG}` : undefined - basePath MUST be set at build time via --build-arg APP_SLUG (the deploy system does this automatically) - Do NOT put AIHUB_AUTH_URL, AIHUB_LOGIN_URL, or other server-side env vars in next.config.ts env{} — that block bakes values at build time, but these vars are only available at container runtime. Server-side code reads process.env directly. - next.config.ts env{} should ONLY contain NEXT_PUBLIC_* values and APP_SLUG - For Prisma apps: COPY prisma/ BEFORE npm ci in the Dockerfile (postinstall runs prisma generate which needs the schema). Do this in BOTH the deps and runner stages. - Include an initial migration that creates all tables (prisma migrate diff --from-empty --to-schema-datamodel prisma/schema.prisma --script). Without it, fresh deploys fail because ALTER TABLE runs before CREATE TABLE. - If using @sparticuz/chromium or Puppeteer for PDF generation, the Dockerfile runner stage needs system libs: libgbm1 libnss3 libgtk-3-0 libatk-bridge2.0-0 fonts-liberation and related deps. Express / FastAPI: - Mount routes at the root (/) — nginx handles the prefix - Use relative asset paths in HTML Streamlit: - No changes needed — the platform auth proxy handles everything - Check the "This is a Streamlit app" box when submitting 5. Make sure the app listens on the PORT environment variable (not hardcoded) and binds to 0.0.0.0 6. Add a Dockerfile. REQUIRED for deployment. Examples: Python: FROM python:3.12-slim WORKDIR /app COPY requirements.txt ./ RUN pip install --no-cache-dir -r requirements.txt COPY . . CMD ["python", "server.py"] Node.js: FROM node:20-alpine WORKDIR /app COPY package.json ./ RUN npm install --omit=dev COPY . . CMD ["node", "server.js"] Make sure CMD matches your entry point and the Dockerfile installs ALL dependencies. 7. If the app needs a database: - The platform auto-provisions a scoped PostgreSQL user per app (full CRUD + DDL for migrations). The DATABASE_URL env var is set automatically — Prisma, SQLAlchemy, etc. read it out of the box. - If you need the shared Nest MySQL (student data): read DB_HOST, DB_USER, DB_PASSWORD - If you need the shared Acquisition PostgreSQL (sales/marketing/audit): read ACQ_DB_HOST, ACQ_DB_USER, ACQ_DB_PASSWORD EXPLICITLY — do NOT use generic "DB_HOST" or you'll connect to the wrong database. - NEVER hardcode credentials. NEVER put database URLs in next.config.ts env{}. 8. Create .env.example with: APP_SLUG=my-app AIHUB_DEV_EMAIL=your-email@egelloc.com PORT=3000 9. Push to a GitHub repo under egelloc-corporation org (private is fine — deploy system has access) 10. ▶ How deployment works Workflow
First time: Submit your app below → an admin approves it → it auto-deploys and appears in the launcher.

After that: Just push to GitHub. Your app auto-redeploys on every push to the main branch — no admin action needed.

One-time setup for auto-deploy: After your app is approved, add a webhook to your GitHub repo:
Go to your repo → Settings → Webhooks → Add webhook

Content typeapplication/json
SecretAsk an admin for the webhook secret
EventsJust the push event
That's it. Push to main and your changes go live automatically.
▶ Common deployment gotchas — read this if your app isn't rendering correctly Troubleshooting
What the platform does automatically
  • Assigns a free port (you never pick one)
  • Provisions a scoped PostgreSQL database + user per app (full CRUD + DDL)
  • Sets DATABASE_URL, PORT, APP_SLUG, and platform env vars
  • Generates nginx routing config
  • Auto-deploys on every push to main (after webhook is set up)
  • Preserves your .env across redeploys
Pre-push checklist
Run through this before your first deploy:
  • ☐ App listens on PORT env var (not hardcoded) and binds to 0.0.0.0
  • ☐ /health endpoint returns {"status":"ok"}
  • ☐ Dockerfile exists and installs all dependencies
  • ☐ No absolute paths (href="/static/...", fetch("/api/...")) — use relative paths
  • ☐ Next.js only: basePath set from APP_SLUG, server-side env vars NOT in next.config.ts env{}
  • ☐ Prisma only: COPY prisma/ before npm ci in Dockerfile, initial migration included
  • ☐ <script src="/hub-navbar.js" defer></script> before </body>
  • ☐ Auth SDK wired in (aihub-auth.js or aihub_auth.py)
1. Path prefix — #1 cause of broken deploys
Symptom: page loads but CSS/JS don't, or “This page isn't working”, or dashboard renders with no data.

Your app is at /slug/ but nginx strips the prefix — the app sees /. Any URL in HTML/JS that starts with / goes to the platform root, not your app.
  • Flask / Jinja: href="static/style.css" and fetch("api/data") (relative, no leading slash)
  • Next.js: set basePath from APP_SLUG in next.config.ts (deploy passes --build-arg APP_SLUG)
  • Streamlit: no change needed
2. Next.js: don't bake server env vars at build time
Symptom: 401 on all API routes, “ECONNREFUSED 127.0.0.1:5051” in logs, auth never works.

next.config.ts env{} bakes values at build time. AIHUB_AUTH_URL, DATABASE_URL, and other server vars are empty during docker build (only set at runtime). Baking "" overrides the real runtime value.

Rule: only put NEXT_PUBLIC_* and APP_SLUG in env{}. Server code reads process.env directly.
3. Database — read the right env vars
Symptom: “connection failed”, “SSL negotiation” error, or wrong data.

Three database options:
  • Auto-provisioned Postgres (default) — DATABASE_URL set automatically. Prisma, SQLAlchemy, etc. read it out of the box.
  • Nest MySQL (student data) — read DB_HOST, DB_USER, DB_PASSWORD
  • Acquisition Postgres (sales/marketing) — read ACQ_DB_HOST, ACQ_DB_USER, ACQ_DB_PASSWORD explicitly
Do NOT use generic DB_HOST if you need Acquisition — you'll connect to MySQL instead of Postgres.
4. Prisma apps — two extra steps
Symptom: build fails with “schema.prisma: file not found”, or container crashloops with migration errors.

  • Dockerfile: add COPY prisma ./prisma BEFORE RUN npm ci in both deps and runner stages (postinstall runs prisma generate)
  • Migrations: include an initial migration that creates all tables: npx prisma migrate diff --from-empty --to-schema-datamodel prisma/schema.prisma --script. Without it, the first deploy fails because ALTER TABLE runs before CREATE TABLE.
